# ReceiptHub v1.7.2 最終運用フロー

## AI判定と再提出の最終判断

AIは、カード番号・利用者名（宛名）・サービスと払先・領収書発行月などに明確な不一致候補を見つけても、領収書ファイルを自動削除せず、再提出依頼も自動作成しません。

```text
AI検査
  ↓
不一致候補あり
  ↓
領収書と実ファイルを保持
  ↓
管理者画面で「再提出判断待ち」として表示
  ↓
管理者が領収書本体を確認して最終判断
```

AIの候補理由は、管理者だけが確認できるメモとして保存します。ユーザー画面にはAIの内部判定を表示しません。

## 管理者が再提出不要と判断する場合

管理者は領収書確認画面で、領収書本体とAI抽出結果を確認します。問題がないと判断した場合は、必要なチェックを補正し、全8項目へチェックを入れて次を実行します。

```text
再提出不要として確認済みにする
```

この操作では次の状態になります。

- 領収書ファイルを保持
- 再提出依頼を作成しない
- 管理者確認者・確認日時・確認メモを保存
- AIが再提出候補とした監査メモは保持
- 「再提出判断待ち」の集計から除外

## 管理者が再提出必要と判断する場合

領収書確認画面で実際に問題があると判断した場合だけ、次を実行します。

```text
再提出を依頼
```

任意でユーザーへ伝える理由を入力できます。理由を空欄にした場合は、AI候補理由または未確認項目を基にシステムが理由を補完します。

再提出依頼を実行すると、次の処理を行います。

1. 再提出依頼レコードとユーザー向け通知を作成
2. 対象領収書レコードを削除
3. Railway Volume上の対象ファイルを削除
4. 提出済み月を「未提出領収書あり」へ戻す
5. ユーザーの対象月画面へ再提出依頼を表示
6. ユーザーが再アップロードすると依頼を対応済みに変更

提出履歴やユーザー別月次一覧から直接再提出依頼を出すボタンは表示せず、原則として領収書確認画面を開いてから判断します。

## 再提出依頼の削除

管理者の「提出履歴」に「再提出依頼管理」を表示します。次の削除方法を利用できます。

- 1件ずつ「依頼を削除」
- 表示中の月の依頼を一括削除
- 未対応・対応済みを含む全期間の依頼を一括削除

依頼を削除すると、ユーザー画面の再提出通知と履歴も削除されます。

> 注意: 再提出依頼を作成した時点で削除された元の領収書レコードと実ファイルは、依頼を削除しても復元されません。

既存の再提出依頼を一度すべて消す場合は、次の操作を行います。

```text
提出履歴
  ↓
再提出依頼管理
  ↓
「全期間・全状態の依頼を削除することを確認しました」にチェック
  ↓
全期間の依頼をすべて削除
  ↓
確認ダイアログで実行
```

## 権限

再提出判断、再提出依頼、依頼削除は管理者ユーザーだけが実行できます。一般ユーザーは、自分に送られた未対応の再提出依頼の確認と、正しい領収書の再アップロードだけを行います。

## 一般ユーザーの提出履歴

提出履歴では、サービス登録数を次の2区分に分けて表示します。

- 必須提出（サブスク、一回払い、支払い種別「その他」の登録サービス）
- 利用時提出（従量課金 / API）

従量課金 / APIは、領収書を提出した場合だけでなく「この月は利用なし」を申告した場合も対応済みとして数えます。進捗はファイル数ではなく「対応済みサービス数 / サービス登録数」と「未対応サービス数」で表示します。同じサービスへ複数の領収書ファイルを追加しても、対応済みサービス数は1件です。提出履歴一覧には混乱を避けるためファイル単位の領収書数を表示しません。
