# ReceiptHub v1.3.1 最終運用フロー

## 管理画面の構成

### 利用サービス管理

「利用サービス管理」は、次の2つをタブで切り替えます。

1. **サービスマスター**
   - サービス名、支払い種別、カード明細・払先候補、状態を管理
   - 25件単位のページ表示
2. **ユーザー別サービス状況**
   - ユーザーをプルダウンで選択
   - 登録状況 → 登録サービス一覧 → 新規登録/停止の順で表示
   - 大量データは縦横スクロールで確認

### ユーザー管理

新規ユーザー作成、登録アカウント、パスワード、権限、利用ステータスを管理します。登録アカウント表の関連画面は「月別状況」だけとし、サービス登録は「利用サービス管理」のユーザー別タブに集約します。

スーパーアカウントの連絡先メール設定は、ページ最下部に表示します。

## 会社共通のご利用代金明細照合

### 1. 全社明細をアップロード

管理者は「ご利用代金明細」ページで対象月を選び、会社全体のカード明細をアップロードします。ユーザーは選択しません。

対象月は利用日の月ではなく、請求・支払対象月です。例えば、5月に利用し6月29日に支払う明細は、2026年6月分として照合します。

### 2. AIが明細表の全行を抽出

OpenAI APIは次を抽出します。

- カード末尾4桁
- 請求・支払対象月
- 支払日
- 明細番号
- 利用日
- ご利用先
- 円請求額
- 外貨金額・通貨
- サービスマスター候補
- 領収書が必要か

AIは明細書からユーザーを直接決めません。明細書は会社共通であり、ユーザーの特定は提出済み領収書との照合結果から行います。

### 3. 全ユーザーの領収書と一対一照合

対象月に一般ユーザー全員がアップロードした領収書を横断し、次を組み合わせて明細行と照合します。

- サービスマスター
- カード明細・払先候補
- 領収書から抽出した払先
- 金額・通貨
- 利用日 / 領収書日付
- その他領収書のメモ

同じ領収書を複数の明細行へ使い回しません。同じサービスを複数ユーザーが利用していて候補が拮抗する場合は、自動でユーザーを確定せず手動確認対象にします。

### 4. 管理者画面の表示

- **黄色**: 領収書が必要だが、全ユーザーの提出物から一致する領収書が見つからない
- **薄橙色**: 対応ユーザーまたはサービスが曖昧で、管理者の確認が必要
- **通常表示**: 対応ユーザー・サービス・領収書が確認済み、または対象外

管理者は各行について、ユーザー / サービスの手動指定、領収書必要・不要の変更、対象外指定ができます。

### 5. 新しい領収書が届いた後

ユーザーが新しい領収書をアップロードすると、保存済みの明細行と自動で再照合します。管理者は「最新の領収書と再照合」を押して、OpenAI APIを再実行せず照合だけを更新することもできます。

APIサービスを「当月利用なし」と申告したユーザーに該当請求が見つかった場合は、申告を取り消して確認対象へ戻します。

## 照合結果PDFとSlack共有

ご利用代金明細書のAI解析と全ユーザー領収書との照合が完了すると、管理者画面の明細書カードに「照合結果PDF」が表示されます。AI解析中または照合結果がない場合はダウンロードできません。

PDFには次を掲載します。

- 対象月、カード末尾4桁、支払日、解析・照合日時
- 未提出・確認対象の詳細
- 全明細照合結果

PDFには、生成日時・`Slack共有用PDF` 表記、件数集計、解析メモ、ユーザー別要対応サマリー、共有先権限に関する注意文を掲載しません。未提出行は黄色、確認対象行は薄橙色で表示します。PDFはA4横向きで、明細数に応じて複数ページになります。ファイル名は `ReceiptHub_YYYY-MM_ご利用代金明細照合結果_ID.pdf` です。

Slackへの自動投稿は行いません。管理者がPDFをダウンロードし、Slackへ手動で共有します。

## データ保持

- 領収書ファイルとカード明細ファイルは最大3ヶ月保存
- 保存期限後は物理ファイルを削除し、監査用メタデータを保持
- ユーザー削除時、会社共通のカード明細自体は削除せず、ユーザー紐付けだけを解除
