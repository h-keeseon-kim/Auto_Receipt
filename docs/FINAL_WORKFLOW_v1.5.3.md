# ReceiptHub v1.5.3 最終運用フロー

## 月の定義

ReceiptHubでは、提出月の前月を対象領収書月として扱います。

```text
2026年07月提出 → 2026年06月分の領収書
2026年01月提出 → 2025年12月分の領収書
```

## ユーザーの提出

1. ユーザーはアップロード画面で提出月を選択します。
2. 画面に対象領収書月が明示されます。
3. 7月提出では6月分の領収書をアップロードします。
4. 従量課金 / APIを6月に利用していない場合は「対象領収書月は利用なし」を選択します。
5. 対象領収書月のサービスがすべて解決したら提出します。

## AI確認

- AIは領収書日付を提出月ではなく対象領収書月と比較します。
- 7月提出・6月領収書は正常です。
- 7月提出に対して5月または7月の領収書は月不一致として扱います。
- 月不一致が明確な場合は従来どおり提出項目から取り下げ、再提出依頼を作成します。
- 日付を確認できない場合は自動削除せず、管理者確認対象にします。

## ご利用代金明細

ご利用代金明細は領収書月で管理します。6月分の明細は、7月提出サイクルにある全ユーザーの6月領収書と照合します。

## 利用停止

最終領収書月を6月としてサービスを停止した場合、そのサービスは7月提出まで選択できます。8月提出からは選択できません。

## リマインダーメール

`{target_month}` は提出月、`{receipt_month}` はその前月の対象領収書月です。管理者が編集していない既定テンプレートはマイグレーションで新しい表記へ更新します。

## 既存データ

マイグレーション `receipts.0020_submission_receipt_month_semantics` は、保存されているAI抽出日付またはAI判定月を使い、既存の月判定を再計算します。旧版の誤判定で物理ファイルまで削除済みの領収書は復元できないため、該当ファイルは再アップロードしてください。

## バージョン

リポジトリ直下の `VERSION` に `1.5.3` を記録します。Web画面には表示しません。


## 領収書プレビューのセキュリティ

管理者の領収書確認画面は、認証済みの同一オリジンからだけ領収書ファイルをiframe表示できます。プレビュー配信レスポンスは `X-Frame-Options: SAMEORIGIN` とし、ReceiptHub以外のサイトからの埋め込みは許可しません。その他の画面は `DENY` のままです。
