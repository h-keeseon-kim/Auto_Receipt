# ReceiptHub v1.6.0 最終運用フロー

## 月の定義

ReceiptHubでは、提出月と全社ご利用代金明細月を同じ月次サイクルとして扱い、その前月を対象領収書月とします。

```text
2026年07月提出 / 2026年07月分明細 → 2026年06月分領収書
2026年01月提出 / 2026年01月分明細 → 2025年12月分領収書
```

## ユーザーの提出

1. ユーザーはアップロード画面で提出月を選択します。
2. 画面に対象領収書月が明示されます。
3. 7月提出では6月分の領収書をアップロードします。
4. 従量課金 / APIを6月に利用していない場合は「対象領収書月は利用なし」を選択します。
5. 対象領収書月のサービスがすべて解決したら提出します。

## AI確認

- AIは領収書日付を提出月ではなく対象領収書月と比較します。
- 7月提出・6月領収書は正常です。
- 7月提出に対して5月または7月の領収書は月不一致として扱います。
- 明確な不一致は再提出対象、確認不能は管理者確認対象です。

## ご利用代金明細

管理者が選択する月は「明細月」です。明細月はユーザーの提出月と同じ月次サイクルを表します。

```text
2026年07月分の全社明細
  → 2026年07月提出サイクルを検索
  → 2026年06月分の全ユーザー領収書と一対一照合
  → 2026年08月提出サイクルは検索しない
```

カード明細の利用日は通常、明細・支払月の前月です。例えば5月利用・6月29日支払いの明細は6月分明細として、6月提出サイクルに登録された5月分領収書と照合します。

新しい領収書が同じ提出月へ追加されると、OpenAI APIを再実行せず、保存済み明細行とのローカル照合を自動更新します。

## 既存明細の移行

マイグレーション `receipts.0021_card_statement_month_semantics` は次を行います。

- 表示名を「領収書月」から「明細月」へ変更
- 旧画面で前月を選んでいた明細を、AI判定明細月が翌月の場合に限り補正
- 翌月提出の領収書へ付いていた旧自動照合を解除
- 管理者が明細画面を開いた際、保存済み明細行を正しい提出月で一度だけ再照合
- 管理者が手動確定したユーザー・サービス指定は維持

OpenAI APIへ明細書を再送信する必要はありません。

## 複数候補による明細照合

明細1行と領収書1件を単一のサービス名だけで結び付けず、金額、通貨、日付、ご利用先、領収書払先、サービスマスターの払先候補を組み合わせて最大5件の候補を作ります。

```text
ご利用先は不一致 / 49.92 USDは完全一致 / 日付一致
  → 領収書ありとして割り当て
  → 薄橙色で管理者確認

22 USDの領収書候補が複数
  → 自動では割り当てない
  → 黄色のまま候補一覧を表示
```

1つの領収書を複数明細へ使い回さず、明細側と領収書側の双方で候補が一意かを確認します。管理者は候補一覧から「この領収書で確定」を選択できます。
