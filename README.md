# Auto Receipt

領収書管理アプリのMVPです。ユーザーは毎月、利用中サービスの領収書をアップロードし、確認後に提出できます。管理者はユーザー発行、サービスマスター管理、ユーザー別サービス登録、ユーザーによる新規利用登録・利用停止の把握、提出履歴での月別提出状況確認、領収書ファイル削除、領収書ファイルと `manifest.csv` のZIP一括ダウンロードを行えます。

## 実装済み機能

- ユーザー登録・ログイン・ログアウト
- 管理者による一般ユーザー発行
  - アカウント名はメールアドレス形式
  - 初期パスワードをランダム生成
  - 初回ログイン時にパスワード変更を強制
- 管理者によるサービスマスター登録
  - サブスク
  - 従量課金 / API
  - 一回払い
  - その他
- 管理者によるユーザー別の利用サービス登録
- ユーザーによるサービス利用登録
  - 管理者が登録したサービスマスターから選択
  - 管理者画面に「ユーザー登録」として表示
- ユーザーによる利用停止
  - 最後にアップロードすべき領収書月を選択
  - 管理者画面に「ユーザー停止」と最終領収書月を表示
  - 停止済みサービスは通常の利用サービス項目から除外
  - 最終領収書月まではアップロード選択肢に残る
- 月別の提出箱
- PDF / PNG / JPG / JPEG / WEBP の領収書アップロード
- バックグラウンドAIによる領収書ファイル名修正
  - ユーザーのアップロード処理は待たせず、後続のCron / 管理コマンドで確認
  - カード末尾4桁が `7210` か確認
  - 領収書上の実際の払先、日付、金額、通貨を抽出
  - 例: `260619_金_OpenAI_220_USD.pdf`
  - 作成困難な場合は管理者向けメモとして提出履歴に表示
  - 管理者向けメモは一般ユーザーには非表示
- バックグラウンドAIによる領収書の提出月チェック
  - AIで抽出した領収書日付が提出月と一致するか確認
  - 異なる月の領収書と判定された場合もユーザー側からは自動削除せず、管理者向けメモに記録
  - 管理者が確認後、必要に応じてユーザーへ再アップロードを依頼
- 提出後の編集ロック
- ユーザー自身の提出履歴・提出詳細確認
- 管理者向け提出履歴
  - ユーザー名順の月別提出状況確認
  - 各ユーザーがアップロードした領収書ファイルの確認
  - 領収書ファイルと履歴の削除
  - 月別ZIPダウンロード
  - ユーザー別ZIPダウンロード
  - ZIP内に監査用 `manifest.csv` を同梱
- 管理者向け利用サービス管理
  - サービスマスター登録・編集・停止・再開
  - サービスマスター一覧のページ表示
  - ユーザー選択式の登録状況 / 登録サービス一覧 / 新規登録・停止確認
  - ユーザーごとの利用サービス登録・編集・停止・再開
- アップロードファイルの最大3ヶ月保存
  - ファイル本体は自動削除
  - サービス名・金額・提出月などのメタデータは保持

## ローカル起動

```bash
cd auto_receipt
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
python manage.py migrate
python manage.py createsuperuser
python manage.py runserver
```

ブラウザで `http://127.0.0.1:8000/` を開きます。

## 管理者によるユーザー発行

一般ユーザーを管理者が発行する運用の場合は、Railwayの環境変数で `ALLOW_SIGNUP=False` を推奨します。

1. 管理者アカウントでログインします。
2. 画面上部の「ユーザー発行」を開きます。
3. 新しく登録するユーザー名としてメールアドレスを入力します。
4. 「パスワード生成」を押します。
5. 画面に表示されたアカウント名と初期パスワードを対象ユーザーへ伝えます。
6. ユーザーは初回ログイン後、強制的にパスワード変更ページへ移動します。
7. パスワード変更後に領収書管理機能を利用できます。

初期パスワードはハッシュ化して保存されるため、後から管理者が再確認することはできません。表示されたタイミングで必ず控えてください。

## サービスマスターと利用サービス登録

サービスは、管理者が登録する「サービスマスター」と、ユーザーごとの「利用サービス」に分かれます。ユーザーは管理者が登録したサービスマスターの中から、新しく使い始めたサービスを自分で利用登録できます。

サービスマスターは「サービス名 + 支払い種別」の組み合わせで管理します。そのため、同じ `ChatGPT` でも `ChatGPT（サブスク）` と `ChatGPT（従量課金 / API）` を別サービスとして登録できます。ユーザー向けの選択肢や一覧では、サービス名と支払い種別を組み合わせた表示名を使います。

### 管理者がサービスマスターを登録する

1. 管理者アカウントでログインします。
2. 画面上部の「利用サービス管理」を開きます。
3. 「サービスマスター登録」を押します。
4. サービス名、支払い種別、メモを入力して登録します。
5. 同じサービス名でも支払い種別が異なれば、別のサービスマスターとして登録できます。
6. 登録後、一般ユーザーの「サービス利用登録」画面に `サービス名（支払い種別）` 形式で表示されます。

### 管理者がユーザーへ利用サービスを登録する

1. 管理者アカウントでログインします。
2. 「利用サービス管理」を開きます。
3. 対象ユーザーをプルダウンで選択します。
4. 「サービスを登録」を押します。
5. サービスマスターからサービスを選択して登録します。
6. 登録後、対象ユーザーのアップロード画面にそのサービスが表示されます。

### ユーザーが自分で利用サービスを登録する

1. 一般ユーザーでログインします。
2. 上部メニューの「利用サービス」を開きます。
3. 「サービス利用登録」を押します。
4. 管理者が登録したサービスマスターから、新しく使い始めたサービスを選択します。
5. 登録後、アップロード画面の利用サービス一覧と領収書アップロードの選択肢に表示されます。
6. 管理者画面には「ユーザー登録」として表示されます。

### ユーザーが利用停止する

1. 一般ユーザーでログインします。
2. 「利用サービス」を開きます。
3. 利用中サービスの「利用停止」を押します。
4. 最後にアップロードすべき領収書月を選択します。
5. 利用停止後、そのサービスは通常の利用サービス項目から外れます。
6. 選択した最終領収書月までは、領収書アップロード時の選択肢に残ります。
7. 管理者画面には「ユーザー停止」と最終領収書月が表示されます。

過去に提出済みの領収書には、提出時点のサービス名・支払い種別がスナップショットとして残ります。

## GPT APIによる領収書ファイル名修正

ユーザーが領収書をアップロード、または提出後にファイル修正した時点では、API判定は実行せず、領収書はまず保存します。OpenAI APIによる確認は、後続のバックグラウンド処理で実行します。

手動実行は以下です。

```bash
python manage.py process_pending_receipts --limit 50
```

Railwayでは、`railway-ai-cron.toml` を参考にCron専用サービスを追加し、定期的に `process_pending_receipts` を実行してください。

処理順は以下です。

1. 領収書内の支払カード末尾4桁が `RECEIPT_CARD_LAST4`、標準では `7210`、と一致するか確認します。
2. 領収書上の実際の払先・販売者・請求元を抽出します。登録サービス名は参考情報だけで、ファイル名には領収書内の払先を使います。
   - 例: `ChatGPT（サブスク）` の払先は `OpenAI` になり得ます。
   - 例: `Claude（サブスク）` の払先は `Anthropic` になり得ます。
3. 支払日または領収書日付、合計金額、通貨を抽出します。
4. すべて高い確度で確認できた場合、`YYMMDD_曜日_払先_金額_通貨.ext` 形式の表示ファイル名を保存します。

例:

```text
260619_金_OpenAI_220_USD.pdf
```

カード末尾、払先、日付、金額、通貨のいずれかが確認できない場合や、AIが作成困難と判断した場合は、ファイル名は元ファイル名のままにし、管理者向けのAIメモを保存します。このメモは管理者の提出履歴・提出詳細・ZIP内 `manifest.csv` には表示されますが、一般ユーザー画面には表示されません。

### 提出月チェック

AIで抽出した領収書日付の年月と、画面で選択している提出月を比較します。

- 一致する場合は、提出月確認ステータスを `当月確認済み` として保存します。
- 異なる場合は、提出月確認ステータスを `提出月不一致` として保存し、管理者向けメモに「ユーザーへ再アップロードを依頼してください」と記録します。
- 異なる月の領収書と判定されても、アップロード済みファイルと履歴は自動削除しません。管理者が内容を確認し、必要に応じてユーザーへ連絡します。
- 日付を確認できない場合は `確認不可` として管理者画面に表示します。

## 期限切れファイル削除

ファイル本体はアップロード時刻から最大3ヶ月後に削除対象になります。手動実行は以下です。

```bash
python manage.py purge_expired_receipts --dry-run
python manage.py purge_expired_receipts --noinput
```

削除後も `Receipt` レコードは残り、提出月・ユーザー・サービス名・金額・元ファイル名・削除日時を確認できます。

## Railway デプロイ手順

1. GitHubにこのリポジトリをpushします。
2. Railwayで `Deploy from GitHub repo` を選びます。
3. PostgreSQLサービスを追加します。
4. アプリサービスに環境変数を設定します。
   - `SECRET_KEY`: Django用の長いランダム文字列
   - `DEBUG`: `False`
   - `ALLOWED_HOSTS`: Railwayのドメイン。例: `your-app.up.railway.app`。`healthcheck.railway.app` と `RAILWAY_PUBLIC_DOMAIN` はアプリ側で自動追加されます。
   - `CSRF_TRUSTED_ORIGINS`: 例: `https://your-app.up.railway.app`。`RAILWAY_PUBLIC_DOMAIN` がある場合は自動追加されます。
   - `DATABASE_URL`: Railway PostgreSQLの接続URL。Railway側で `PGHOST` などを使う構成でも動きます。
   - `RECEIPT_MEDIA_ROOT`: `/app/media`
   - `RECEIPT_RETENTION_MONTHS`: `3`
   - `ALLOW_SIGNUP`: 一般ユーザーの自己登録を許可するなら `True`。管理者がユーザーを作成する運用なら `False`
   - `OPENAI_API_KEY`: OpenAI APIキー
   - `OPENAI_MODEL`: 領収書解析に使うモデル名。Railway上で設定済みの値を使います。
   - `RECEIPT_AI_FILENAME_ENABLED`: AIファイル名修正を有効化するなら `True`
   - `RECEIPT_CARD_LAST4`: 確認対象カード末尾4桁。標準は `7210`
   - `RECEIPT_AI_TIMEOUT`: OpenAI API呼び出しタイムアウト秒数
5. Railway Volumeをアプリサービスに追加し、マウントパスを `/app/media` にします。
6. ヘルスチェックパスは `railway.toml` で `/health/` に設定済みです。
7. 初回デプロイ後、Railway Shell等から管理者を作成します。

一般ユーザーが自分でアカウント作成する運用の場合は、Railwayの環境変数で `ALLOW_SIGNUP=True` を設定してください。`ALLOW_SIGNUP=False` の場合、登録リンクは非表示になり、`/accounts/register/` に直接アクセスしてもログイン画面へ戻ります。

```bash
python manage.py createsuperuser
```

## Railway ヘルスチェック

Railwayのヘルスチェック用に `/health/` を用意しています。このURLはログイン不要でHTTP 200を返し、DBクエリも実行しません。`/accounts/login/` はDjangoのホスト検証やHTTPSリダイレクトの影響を受ける可能性があるため、Railwayでは `/health/` を使います。

## Railway Cron設定

Cronは2種類を想定しています。どちらも同じGitHubリポジトリから追加サービスを作り、Cron専用サービスとして動かします。

### 期限切れファイル削除

- Start Command: `python manage.py purge_expired_receipts --noinput`
- Cron Schedule: `15 18 * * *`
- 共有する環境変数: DB接続情報、`RECEIPT_MEDIA_ROOT=/app/media`
- 設定例: `railway-cron.toml`

### AI領収書確認

- Start Command: `python manage.py process_pending_receipts --limit 50`
- Cron Schedule: `*/10 * * * *`
- 共有する環境変数: DB接続情報、`RECEIPT_MEDIA_ROOT=/app/media`, `OPENAI_API_KEY`, `OPENAI_MODEL`
- 設定例: `railway-ai-cron.toml`

AI確認を手動で再実行したい場合は、以下を使います。

```bash
python manage.py process_pending_receipts --limit 50 --retry-failed
```

## 注意点

MVPでは領収書ファイルをDjangoのローカルストレージに保存します。RailwayではVolumeを `/app/media` にマウントしてください。将来的にファイル数・容量が増える場合は、S3互換ストレージへの移行を推奨します。

## AI確認チェックボックスと再提出指示

`process_pending_receipts` は領収書ごとに以下を確認し、管理者画面の提出履歴にチェックボックスとして表示します。

- カード末尾が `RECEIPT_CARD_LAST4`（デフォルト: `7210`）と一致するか
- 払先を確認できたか
- 登録サービス名と領収書上の払先が同一または合理的に関連しているか
- 日付を確認できたか
- 金額を確認できたか
- 通貨を確認できたか
- 領収書日付が提出月と一致するか

曖昧、確認不可、不一致の項目はチェックを入れず、処理済みで未チェック項目がある領収書は管理者画面でハイライトされます。管理者はハイライトされた領収書を手動確認し、本当に問題がある場合は「再提出指示」ボタンを押せます。

「再提出指示」を押すと、対象の領収書レコードとアップロードファイルを削除し、提出済みだった月は下書きに戻します。ユーザー画面には再提出依頼が表示され、同じ提出月・同じサービスの領収書を再アップロードすると依頼は対応済みに変わります。

追加マイグレーション:

```text
receipts/migrations/0007_receipt_ai_check_amount_receipt_ai_check_card_last4_and_more.py
```
