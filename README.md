# ReceiptHub

**Version: 1.7.1**

ReceiptHubは、社内で利用しているサブスクリプション、従量課金/API、一回払いサービスの領収書を月単位で回収・検査・管理するDjangoアプリです。

バージョンはリポジトリ直下の `VERSION` に記録します。Web画面には表示しません。

## 月の定義

一般ユーザーは、画面上で**領収書発行月**を直接選択します。内部では、その翌月を提出・リマインダー・全社明細照合の月次サイクルとして保持します。

```text
領収書発行月 2026年06月 → 内部提出サイクル 2026年07月
領収書発行月 2025年12月 → 内部提出サイクル 2026年01月
```

ユーザー画面には内部提出サイクルを表示せず、領収書発行月だけを基準に操作します。AIの日付確認、利用停止時の最終領収書月、API利用なし申告、カード明細照合、リマインダーメールは同じ対応関係を使用します。

## v1.7.1で実現する最終フロー

### 領収書プレビュー

管理者の領収書確認画面では、領収書ファイルを同一オリジンのiframe内へ表示します。プレビュー配信エンドポイントだけ `X-Frame-Options: SAMEORIGIN` を返し、ReceiptHub自身の確認画面からの埋め込みを許可します。その他の画面は従来どおりクリックジャッキング対策の `DENY` を維持します。表示できないブラウザー向けに「別タブで開く」も用意しています。

### 一般ユーザー

1. ログインすると「利用サービス」が最初に表示されます。
2. サービスマスターに登録済みのサービスは「サービス利用登録」から選択し、管理者承認なしで利用中サービスへ追加します。
3. サービスマスターに存在しない新規サービスだけ、サービス名・支払い方法・用途を入力して「新規サービス例外申請」を提出します。管理者の承認後にサービスマスターと利用中サービスへ追加されます。
4. 利用しなくなったサービスは、最後に提出すべき領収書月を指定して停止します。
5. 毎月の「アップロード」で、何月に発行された領収書を提出するか「領収書発行月」を直接選択します。登録済みサービスを選び、「＋ 領収書を追加」から1件または複数件のファイルを追加します。同じサービスへ何度でも追加できます。ドラッグ中は「ここでファイルを離してください」を大きく表示し、受付後は件数・ファイル名・アップロード中状態を明示します。
6. 返金・プラン変更・追加請求など登録サービス外の領収書は、サービス欄の「その他」を選び、必須メモ付きで追加します。
7. 従量課金/APIサービスを選択した領収書発行月に利用していない場合は、「この月は利用なし」を申告します。
8. 選択した領収書発行月の全サービスについて、領収書またはAPI利用なし申告が揃うと提出できます。
9. 管理者が「AIでファイル名を修正・検査」を実行すると、未処理の領収書だけが検査されます。
10. AIが不一致候補または確認不能項目を検出しても、領収書は自動削除されず、管理者の確認対象として保持されます。
11. 管理者が領収書本体を確認して問題があると判断した場合だけ再提出依頼が作成され、対象領収書が提出項目から取り下げられます。

### 管理者ユーザー

1. 「提出履歴」で対象月のユーザー別進捗を確認します。
2. ユーザーがファイルをアップロードできない場合は、ユーザー別月次ページの「管理者代理アップロード」から対象サービスとファイルを選び、本人に代わって登録します。代理アップロード欄のユーザー選択・前後移動から、提出履歴へ戻らず別ユーザーへ連続して切り替えられます。
3. 代理アップロード後は「確認」から領収書本体をプレビューし、AI検査、確認チェック、表示ファイル名の補正、管理者確認済みの確定まで行います。
4. 未提出サービス、API利用確認待ち、AI要確認、再提出判断待ち、再提出待ちを一覧で把握します。
5. 「AIでファイル名を修正・検査」を押すと、未処理の領収書だけをバックグラウンド処理します。
6. 処理中は「AI抽出中」、完了した領収書は完了順に画面へ反映されます。
7. AIが問題候補とした領収書はハイライトされます。管理者が原本を確認し、問題なければ「再提出不要として確認済み」、問題があれば「再提出を依頼」を選択します。
8. ユーザー別月次ページで、利用サービスごとの領収書提出状況を確認できます。
9. 「ご利用代金明細」で会社全体のカード明細を1件アップロードすると、AIが全明細行を抽出し、明細月と同じ提出サイクルに保存された前月分の全ユーザー領収書を一括照合します。例えば7月分明細は、7月提出サイクルの6月分領収書と照合します。
10. 各明細行について、金額・通貨・日付・ご利用先・サービスから最大5件の領収書候補を作成します。ご利用先が一致しなくても金額が一意に一致すれば提出済みとして扱い、関係確認のため薄橙色で表示します。同額候補が複数ある場合は自動確定せず黄色で残します。
11. 解析・照合完了後に「照合結果PDF」をダウンロードし、権限を限定したSlackチャンネルへ共有できます。
12. 「サービス例外申請」で、サービスマスターに存在しない新規サービスの申請だけを確認し、承認または却下します。承認時は新しいサービスマスターと対象ユーザーの利用サービスへ反映されます。
13. 「利用サービス管理」は「サービスマスター」と「ユーザー別サービス状況」をタブで切り替えて管理します。
14. 「ユーザー管理」でユーザー追加・削除・ステータス・パスワードを管理します。スーパーアカウント設定はページ最下部に表示します。
15. 「提出履歴」の再提出依頼管理から、依頼を1件ずつ削除、表示月単位で一括削除、または全期間・全状態の依頼を一括削除できます。
16. 「メール送信」でリマインダー日、件名、本文、テスト送信を管理します。

### スーパーアカウント

- 一般ユーザーと管理者ユーザーの権限を切り替えられます。
- 管理者ユーザーの作成・削除・パスワード変更も行えます。
- 一般の管理者ユーザーは、他の管理者ユーザーの権限・削除・パスワードを操作できません。

## 実装済み機能

### アカウント・権限

- Django認証によるログイン・ログアウト
- 管理者によるメール形式アカウント発行
- ランダム初期パスワード生成
- 初回ログイン時および管理者による再発行後のパスワード変更強制
- パスワードのランダム再発行・管理者指定パスワードへの変更
- 一般ユーザー / 管理者ユーザーの権限分離
- スーパーアカウントだけが権限変更可能
- ユーザー削除時に、関連サービス、提出履歴、領収書ファイルを削除（会社共通のカード明細は監査履歴として保持し、ユーザー紐付けだけ解除）
- ユーザーステータス「利用中 / 停止中」
- 利用中サービスが0件になると自動で停止中、1件以上になると自動で利用中

既存パスワードはハッシュ化されているため復元表示できません。新規発行または再発行したパスワードだけを、その場で一度表示します。

### サービス管理

- 管理者がサービスマスターを登録
- 支払い種別: サブスク / 従量課金・API / 一回払い / その他
- `サービス名 + 支払い種別` で一意
- 例: `ChatGPT（サブスク）` と `ChatGPT（従量課金 / API）` を別登録可能
- カード明細・払先の表記候補を登録可能
  - 例: `OPENAI *CHATGPT, OPENAI.COM`
- 管理者によるユーザー別サービス登録
- 一般ユーザーによる、サービスマスターからの直接利用登録
- サービスマスターに存在しない場合だけ利用できる新規サービス例外申請（サービス名・支払い方法・用途）
- 管理者画面の未対応通知、承認・却下、承認時のサービスマスター自動作成
- 承認前は利用サービスへ追加せず、承認後に「例外申請承認」として対象ユーザーへ登録
- 一般ユーザーによる利用停止と最終領収書月の指定
- サービス利用状況に応じたユーザーステータス自動同期

### サービス例外申請フロー

サービスマスターに登録済みのサービスは、一般ユーザーが「サービス利用登録」から直接選択します。この操作は例外申請ではなく、管理者承認も必要ありません。

「サービス利用登録」の一覧に存在しない新規サービスを使いたい場合だけ、次の内容で例外申請を提出します。

- サービス名
- 支払い方法（サブスク / 従量課金・API / 一回払い / その他）
- 用途

申請直後の状態は「確認待ち」で、利用サービスや領収書アップロードの選択肢には追加されません。管理者には上部メニューの件数バッジと画面内通知が表示されます。

申請画面で入力したサービス名・支払い方法がすでにサービスマスターに存在する場合は、例外申請を受け付けず「サービス利用登録」へ案内します。停止中のマスターが存在する場合は、管理者への再開依頼を案内します。

管理者が承認すると、新しいサービスマスターを自動作成し、その後、対象ユーザーへ利用サービスを登録してユーザーステータスも同期します。申請後に同じマスターが別経路で登録された場合だけ、重複作成を避けて既存マスターを利用します。却下時は理由の入力を必須とし、ユーザーは利用サービス画面の申請履歴から確認できます。

同じユーザー・同じサービス名・同じ支払い方法の確認待ち申請は重複提出できません。同じ名前でも支払い方法が異なる申請は別申請として扱います。

### 一般ユーザーの月次画面UX

- 「サービス利用登録」の一覧に目的のサービスがない場合は、同じ画面内の青い案内枠から「新規サービス例外申請」へ進む
- 例外申請画面には「サービス利用登録へ戻る」を常設
- アップロード画面では「領収書発行月」だけを選択し、内部の提出月は表示しない
- 未完了の状態は「下書き」ではなく「未提出領収書あり」と表示
- 一部サービスが残っていてもアップロード済み領収書は保存し、黄色の注意文で残りを案内
- 提出履歴では、領収書発行月、サービス登録数、確認済み数、未提出・未確認数を表示
- ユーザー画面ではファイル保存期限を表示しない

### 月次領収書提出

- PDF / PNG / JPG / JPEG / WEBP対応
- 領収書・管理者代理アップロード・ファイル差し替え・全社ご利用代金明細書でドラッグ＆ドロップ対応
- 複数回に分けたアップロード
- サービス選択後に「＋ 領収書を追加」を表示し、ファイル選択時に自動アップロード
- 1回のファイル選択で複数ファイルを追加可能
- 同じサービスへ追加操作を繰り返し、領収書を何件でも登録可能
- 全ユーザーのサービス選択肢に「その他」を常設
- 「その他」は内容メモ必須で、登録サービスの提出済み判定には流用しない
- 提出後に登録サービスまたは「その他」の領収書を追加すると「未提出領収書あり」へ戻し、再提出可能
- 提出後もファイル差し替え可能
- 月別提出履歴・詳細確認
- 従量課金/APIサービスの「この月は利用なし」申告
- 選択した領収書発行月の全サービスが解決するまで提出完了不可
- 管理者が再提出を依頼した領収書は一覧から削除され、再アップロード可能
- 管理者がユーザー別月次ページから、対象ユーザーの登録サービスまたは「その他」を選んで代理アップロード可能
- 管理者代理アップロードはユーザー画面にも即時反映し、通常のAI検査・カード明細照合・再提出フローの対象になる
- 管理者専用の領収書確認画面でPDF・画像をプレビューし、AI結果、8項目チェック、表示ファイル名、管理者メモを確認・補正可能
- すべてのチェックが揃った領収書を「管理者確認済み」として確認者・確認日時付きで記録
- AIは再提出依頼を自動作成せず、管理者確認画面から「再提出不要として確認済み」または「再提出を依頼」を選択可能
- 再提出依頼は個別、表示月単位、全期間単位で削除可能。削除するとユーザー画面の通知も消える
- アップロード元と実行者を監査情報として保存し、画面とZIP内 `manifest.csv` で確認可能
- 提出済み月へ管理者が追加した場合は「未提出領収書あり」へ戻し、ユーザーの再提出対象にする

### OpenAIによる領収書検査

AI処理はユーザーのアップロード中には実行しません。管理者が提出履歴のボタンを押した時だけ、未処理の領収書を検査します。月の一致判定では提出月そのものではなく、その前月に当たる対象領収書月を使用します。

確認項目:

- 支払カード末尾4桁が `RECEIPT_CARD_LAST4` と一致するか
- 領収書上の払先を確認できるか
- 領収書の利用者名・宛名が対象ユーザーと一致するか
- 登録サービスと払先に合理的な関連があるか
- その他領収書では、必須メモと領収書内容に合理的な関連があるか
- 日付を確認できるか
- 金額を確認できるか
- 通貨を確認できるか
- 領収書日付の年月が、提出月の前月に当たる対象領収書月と一致するか

サービス名と払先は完全一致を前提にしません。

- `ChatGPT（サブスク）` と `OpenAI`
- `Claude（サブスク）` と `Anthropic`

のような提供元・請求元の関係を判定します。サービスマスターの「カード明細・払先の表記候補」もAIへ渡します。

「その他」の追加領収書では、ユーザーが入力した必須メモもAIへ渡します。メモは返金・プラン変更などの背景を理解するための参考情報であり、領収書ファイル内の払先、日付、金額、通貨、カード番号を常に優先します。メモと領収書が明確に矛盾する場合も自動取り下げはせず、再提出候補として管理者確認対象にします。

#### 再提出候補と管理者判断

次のようにAIが明確な不一致候補を検出しても、領収書と実ファイルは保持し、再提出依頼を自動作成しません。

- カード末尾4桁が対象カードと違う
- 利用者名・宛名が対象ユーザーと明確に違う
- サービスと払先が明確に無関係
- 「その他」の必須メモと領収書内容が明確に矛盾する
- 領収書日付の年月が対象領収書月と違う

対象領収書には「再提出判断待ち」と理由を表示します。管理者は領収書プレビューを確認し、次のどちらかを選択します。

- 問題なし: チェックを補正し、「再提出不要として確認済みにする」
- 問題あり: 必要に応じてユーザー向け理由を入力し、「再提出を依頼」

再提出を依頼した時点で初めて対象領収書と実ファイルを提出項目から削除し、提出済み月を「未提出領収書あり」へ戻し、ユーザーへ依頼を表示します。

#### 曖昧・確認不能

AIが断定できない項目はチェックを付けず、領収書は保持します。管理者画面でハイライトし、管理者用メモとチェックリストを表示します。一般ユーザーにはAI内部メモを表示しません。

利用者名・宛名は、領収書の `Billed to`、`Bill to`、`Customer`、`Account holder`、宛名、購入者名などを抽出し、対象ユーザーの登録氏名・メールアドレスと照合します。宛名が会社名や部署名だけ、記載がない、または表記だけでは個人を特定できない場合は未チェックのまま管理者が確認します。宛名チェックはファイル名作成可否とは独立しており、宛名だけ曖昧でも他項目が揃えばファイル名候補は保持します。

#### ファイル名

確認できた情報から、表示・ダウンロード用ファイル名を次の形式で生成します。

```text
YYMMDD_ユーザー名_企業名または短い取引ラベル_金額_通貨.拡張子
```

例:

```text
260602_test_Anthropic_220_USD.pdf
```

その他領収書では、領収書本体を優先しつつ必須メモを補助情報にして、例えば次のような短いラベルを生成できます。

```text
260602_test_OpenAI返金_220_USD.pdf
```

ユーザー名部分は、Djangoの姓が設定されていれば姓、未設定ならメールアドレスの `@` より前を使います。内部ストレージ上の物理ファイル名は衝突防止用のUUIDを維持します。

### ご利用代金明細書の全ユーザー照合

管理者は専用の「ご利用代金明細」ページから、会社全体のカード明細を**明細月**ごとにアップロードします。ユーザーは選択しません。AIが明細表の全行を抽出した後、**明細月と同じ提出サイクル**にすべての一般ユーザーがアップロードした**前月分の領収書**を横断して、一対一で照合します。例えば2026年07月分明細は、2026年07月提出サイクルにある2026年06月分領収書と照合し、2026年08月提出の領収書は対象にしません。

AIは次を抽出します。

- カード末尾4桁
- 請求・支払対象月
- 支払日
- 明細番号
- 利用日
- ご利用先
- 円請求額
- 外貨額・通貨
- 対応する登録サービス
- 領収書が必要か

明細月は利用日の月ではなく請求・支払月として扱います。例えば5月利用、6月29日支払いの明細は6月分であり、ReceiptHubでは**6月提出サイクルの5月分領収書**と照合します。同様に、7月分明細は7月提出サイクルの6月分領収書と照合します。

照合時は、各明細行と全ユーザー領収書の組み合わせについて次を評価し、最大5件の候補を保存します。

- 外貨金額と通貨の一致
- 円金額の一致または近似
- 明細のご利用先と領収書上の払先の関連
- サービスマスターの払先候補、サービス名、その他メモとの関連
- 利用日と領収書日付の一致または近接

ご利用先が一致しなくても、金額と通貨が一意に一致し、同じ領収書が別の明細行とも競合しない場合は、領収書が提出されたと判定します。ただし金額だけの一致はサービス対応を断定できないため、行を薄橙色の管理者確認対象にします。22 USDや220 USDのような同額請求が複数存在する場合は自動確定せず、候補一覧を表示します。

表示ルール:

- 黄色: 提出済み領収書を一意に確定できていない
- 薄橙色: 金額一致などで領収書の提出は確認できたが、ご利用先・サービスとの対応確認が必要
- 通常表示: 対応するユーザー・サービス・領収書を十分な根拠で確認済み、または管理者が対象外と確認済み

同じサービスの明細行が複数ある場合も、1つの領収書を複数行へ使い回しません。管理者は「照合候補」を開いて、候補ごとのユーザー、サービス、金額、日付、一致根拠を確認し、「この領収書で確定」を選択できます。行ごとの「ユーザー / サービス」手動指定や対象外設定も引き続き利用できます。新しい領収書がアップロードされた後は「最新の領収書と再照合」でOpenAI APIを再実行せず、保存済み明細行との照合だけを更新できます。

APIサービスについてユーザーが「対象領収書月は利用なし」と申告していても、カード明細に該当請求が見つかった場合は申告を取り消し、管理者確認対象へ戻します。

#### 照合結果PDFとSlack共有

AI解析・領収書照合が完了すると、明細書カードに「照合結果PDF」ボタンが表示されます。ダウンロードされるPDFには次を含みます。

- 対象月、解析ステータス、カード末尾4桁、支払日
- アップロード・解析完了・最終照合日時
- 未提出・確認対象の詳細
- 全明細行の照合結果

PDFでは、生成日時・Slack共有用表記、集計テーブル、解析メモ、ユーザー別要対応サマリー、共有先権限に関する注意文を表示しません。黄色は領収書未提出、薄橙色は管理者による確認が必要な行です。PDFは横向きA4・複数ページで生成され、日本語を含む長いサービス名やユーザー名はセル内で折り返します。

Slack APIへの自動投稿は行わず、管理者がダウンロードしたPDFを手動で共有する運用です。

### 管理者ダッシュボード

管理者の開始ページは「提出履歴」です。

- 提出状況はユーザー名順
- アップロード済み領収書はアップロード日時の新しい順
- 未提出ユーザー・未提出サービス数
- API利用確認待ち数
- AI未確認・AI要確認・手動確認対象
- 再提出待ち
- ユーザー別月次状況へのリンク
- ユーザー別月次状況からの管理者代理アップロード
- 代理アップロード済み領収書のプレビュー・AI個別実行・チェック・表示ファイル名修正・確認済み確定
- 領収書の個別ダウンロード・削除
- 提出済みZIPダウンロード
- ZIP内 `manifest.csv`

### リマインダーメール

Resend SMTPを使います。通常リマインダー日と警告日は管理者画面から変更でき、初期値は毎月4日と10日です。

#### 通常リマインダー

- 利用中の一般ユーザーだけが対象
- サブスク、一回払い、その他の未提出サービスがあるユーザーへ送信
- APIだけが未確認の場合は送信しない

#### 重要リマインダー

- 利用中の一般ユーザーだけが対象
- 未提出サービスまたはAPI利用確認待ちがあるユーザーへ送信
- 件名には必ず `【重要】` を付与
- 初期本文には「本日中」の案内を含む
- サブスク領収書は提出済みでもAPIサービスが未確認なら、APIを利用していないか確認するメールを送信

ユーザーがAPIサービスを対象領収書月に利用していない場合は、アップロード画面で「対象領収書月は利用なし」を選ぶと重要リマインダー対象から外れます。

管理者は「メール送信」で次を変更できます。

- 通常リマインダー日
- 警告日
- 通常メール件名・本文
- 重要メール件名・本文

利用可能なテンプレート変数:

```text
{app_name}
{user_name}
{target_month}
{receipt_month}
{upload_url}
{missing_services}
{api_pending_services}
```

送信ログを保存し、同一月・同一ユーザー・同一種別の重複送信を防止します。停止中ユーザーにはリマインダーもテストメールも送りません。

## Railway運用

### Webサービス

- `railway.toml`
- Start Command: `/app/start.sh`
- Volume mount: `/app/media`
- PostgreSQLと同じ `DATABASE_URL`
- OpenAI・SMTP環境変数を設定

`start.sh` はデプロイ時にマイグレーションを適用してからGunicornを起動します。

### メール送信Cron

管理者画面で日付を変更できるため、4日用・10日用を分けず、毎日1回のCronサービスを1つ使います。

```bash
python manage.py send_receipt_reminders --kind auto
```

設定例は `railway-email-reminder-daily.toml` です。Cronは毎日起動し、管理者設定の日だけ送信します。

### AI処理

AI検査用Cron・Workerは不要です。管理者がWeb画面のボタンを押した時だけWebサービス内で実行します。

保守用CLI:

```bash
python manage.py process_pending_receipts --limit 100
```

### 期限切れ削除

期限切れ削除コマンドは、`/app/media` の実ファイルへアクセスできる実行環境で動かしてください。設定例は `railway-cron.toml` です。

## アップグレード

git apply ReceiptHub_v1.7.1.patch
git add .
git commit -m "Release ReceiptHub v1.7.1"

```

v1.7.1では `receipts.0024_receipt_ai_resubmission_recommendation` を追加します。AIの再提出候補フラグと管理者向け理由を保存します。既存の再提出依頼は維持されますが、提出履歴から個別・月単位・全期間単位で削除できます。領収書ファイルはAI判定だけでは削除されません。

```

### スーパーアカウントの連絡先メール

スーパーアカウントのログイン名は `admin` のまま維持し、連絡先メールアドレスは任意にできます。「ユーザー管理」の「スーパーアカウント設定」でメール欄を空欄にして保存すると、そのメールアドレスを一般ユーザーのアカウント名として再利用できます。

CLIから実施する場合は次を実行します。

```bash
python manage.py clear_superuser_email --username admin
```

この操作はスーパーアカウントのログイン名・パスワード・権限を変更しません。
